1Executive Summary

NeuroTrace is Chrome DevTools for AI agents. It is a Python library that wraps an agent's execution loop, silently records a structured event for every decision the agent makes — each model call, each tool invocation, each error and retry, with timing and payloads — saves that recording to a local database, and renders it back as an inspectable timeline.

The business case rests on a single observation: agent frameworks make it easy to build a loop and hard to see inside it. When an agent misbehaves in production — loops on a tool, invents an argument, silently swallows a failure — the standard remedy today is to add print() statements and run it again, hoping the non-deterministic failure reproduces. NeuroTrace captures the run once, in full, so diagnosis becomes reading rather than re-rolling the dice.

What is complete

The capture path is finished end to end, and the data is now reachable by other software. The event schema, the tracer, both storage backends (in-memory and SQLite), the OpenAI-compatible adapter with security validation, the command-line interface, a terminal timeline renderer, and a read-only JSON API server all work and are covered by tests. An agent adopts NeuroTrace by changing one construction line; the loop itself is untouched.

What is not

The graphical HTML timeline viewer — arguably the most demo-able piece — is not built, though the API it will consume now exists. Streaming responses are not captured correctly. There is no redaction of sensitive data, and no async or threaded agent support. These are enumerated honestly in Section 9, because documentation that oversells a pre-release library is worse than none.

The one-sentence pitch

Instrument an agent in one line, and get a complete, causally-ordered, replayable record of everything it did — portable across OpenAI, xAI, Groq, and locally-hosted models, with no vendor SDK dependency and no data leaving the machine.

2The Problem We Are Solving

2.1 What an agent is, and why it is hard to watch

An AI agent is a program that pursues a goal autonomously. Given an instruction — "find the cheapest flight to Chicago" — it does not execute a fixed script. It asks a language model what to do next, calls a tool (a function, an API, a database query), feeds the result back to the model, and repeats. The loop runs until the model declares itself done.

That loop is the product's value and its liability. The value: the agent handles situations nobody wrote code for. The liability: nobody wrote code for those situations, so nobody knows what it did. The developer sees an instruction go in and an answer come out. Everything between is invisible.

2.2 The four failure modes that cost real money

FailureWhat it looks likeBusiness cost
Silent loopThe agent calls the same tool twenty times with slightly different arguments before giving up.Direct token spend, multiplied. Latency the user experiences as a hang. Invisible until the invoice arrives.
Hallucinated callThe model asks for a tool that does not exist, or passes an argument the schema never offered.A crash in production, or worse — a call that succeeds against a parameter you never intended to expose.
Swallowed errorA tool fails; the framework catches it, feeds a vague message back to the model, and the agent carries on to a wrong answer.Wrong output delivered confidently. The most expensive class of failure, because nobody notices.
Non-reproducible bugA user reports a bad run. Re-running the same prompt produces a different, working result.Engineering hours burned on failures that cannot be reproduced on demand. The dominant cost in practice.

2.3 Why existing tooling does not close the gap

Three approaches exist today, and each falls short in a specific way:

  • Raw logging (print / logging). Unstructured and ordered by wall-clock, so causality — this model call requested that tool — is lost. Reconstructing the loop from interleaved text is manual archaeology.
  • Framework-native tracing. LangChain, LlamaIndex, and CrewAI each ship something, all mutually incompatible, all absent if you wrote the loop yourself against a raw SDK. Adopting one couples your observability to your framework choice.
  • Hosted platforms (LangSmith, Langfuse, Braintrust). Capable, but they ship prompts and tool payloads to a third party — a non-starter for regulated data — and introduce a vendor dependency and a per-seat cost at exactly the stage where a team is still deciding whether agents work at all.

The gap NeuroTrace targets: a framework-agnostic, locally-hosted, zero-dependency way to capture an agent run as structured, causally-linked data.

Analogy that lands with non-technical stakeholders

Before browser DevTools, debugging a slow web page meant guessing which request was slow. DevTools did not make pages faster — it made them legible, and the speed followed. NeuroTrace is that intervention, applied to agent loops.

3The Product

3.1 Definition

NeuroTrace is an observability layer for agent execution. It is conceptually adjacent to distributed tracing systems (Jaeger, Honeycomb) — the same span/parent model — but purpose-built for reasoning loops rather than microservice request chains. Where a distributed trace answers "which service was slow?", a NeuroTrace answers "why did the agent decide that?"

3.2 The adoption experience

The design constraint that shaped the entire product: instrumentation must not appear in the agent loop. A developer wraps the client and changes nothing else.

# Before — an ordinary tool-calling loop
client = OpenAI()
response = client.chat.completions.create(model="gpt-4o", messages=messages)

# After — one line changes; the loop below is byte-identical
with Tracer(name="my-agent", storage=SQLiteStorage("traces.db")) as tracer:
    client = trace_openai(OpenAI(), tracer)
    response = client.chat.completions.create(model="gpt-4o", messages=messages)
    messages += client.dispatch_tool_calls(response, {"get_weather": get_weather})

The verified consequence: examples/openai_agent.py contains a complete agent loop with zero NeuroTrace calls inside it. Only the construction line knows tracing exists. This matters commercially — the adoption cost is near zero, and so is the cost of removing it if the team decides against it. Nothing is locked in.

3.3 The output

After a run, the trace is inspected from the terminal. The tree structure is the point: each tool call sits under the model call that requested it, so causality is visible at a glance rather than inferred from timestamps.

$ neurotrace view traces.db
Trace: weather-agent  (d4561a4d)  2026-07-19T18:56:11 -> 2026-07-19T18:56:11
├─ llm_call  gpt-4o  412.7ms
│  └─ tool_call  get_weather  1.2ms
├─ llm_call  gpt-4o  380.1ms
├─ llm_call  gpt-4o  291.5ms
│  └─ tool_call  book_flight  0.1ms
│     └─ error  [error: no tool named 'book_flight']
└─ llm_call  gpt-4o  244.8ms

Read that trace as a stakeholder: four model calls were billed, one tool succeeded, and the agent hallucinated a book_flight capability it was never given. That diagnosis took five seconds and no re-run.

The pipeline from a live agent to a readable timeline. Note the single persistence point — a design decision with a consequence documented in Section 9.

Figure 2 — Data flow. Events accumulate in memory during the run and are written once when the Tracer's context block exits. This keeps the hot path free of disk I/O, at the cost documented in Section 9.3.

4.3 Sequence of one traced agent turn

This is the most important diagram in the document, because it shows the ordering problem the architecture had to solve. The model requests a tool inside a completion, but the tool only runs after that completion has returned and its span has already closed.

Figure 3 — One traced turn. Step 8 is the architectural crux: Tracer.under() re-parents the tool span under a completion span that has already closed, preserving the causal link the timeline depends on.

4.4 Flat storage, tree presentation

Events are stored as a flat list with a parent_id pointer. The tree is rebuilt at read time, not enforced at write time. This is why the event list's order differs from the timeline's shape — spans append when they close, so a parent appears in the list before its children even though it renders above them.

Figure 4 — Flat capture, tree presentation. Deferring tree construction to read time means capture never has to know the final shape, and any future viewer reuses the same _build_children logic rather than re-deriving it.

4.5 Data model

Two tables. A Trace is one agent run; an Event is one thing that happened inside it. The payload column is JSON, which is what makes the schema adapter-agnostic.

TableColumnsPurpose & notes
traces trace_id (PK), name, started_at, ended_at, metadata_json One row per agent run. ended_at null means the run is still in progress or crashed. metadata_json carries free-form run context — the example stores which provider was used, enabling cross-provider comparison.
events event_id (PK), trace_id (FK), event_type, payload_json, timestamp, duration_ms, parent_id, error One row per recorded decision, indexed on trace_id. parent_id is a self-reference that encodes causality. Both tables use upsert, so re-saving an in-progress trace updates rows rather than duplicating them.

5Component Walkthrough

Every file in the library, what it does, and why it earns its place.

core/events.py — the vocabulary (171 lines)

Defines Event, Trace, EventType, and four constructor functions. This was the first thing built, and deliberately so: every other component — recorder, storage, adapters, viewer — has to agree on what an "event" is, so the schema is the contract that makes parallel development possible.

Business value: a stable schema is what allows a second adapter (Anthropic, LangChain) or a second consumer (a dashboard, an export to OpenTelemetry) to be added later without renegotiating the data format. It is the asset with the longest half-life in the codebase.

core/tracer.py — the recorder (167 lines)

A context manager owning one Trace per run, with nested llm_call and tool_call spans that time themselves and infer parentage from a stack. Two details matter disproportionately:

  • Pre-generated event IDs. A span's ID is created on entry, before the event object exists, so children can reference a parent that has not finished yet.
  • __exit__ always persists — including when the agent raised. A crashed run still leaves a partial trace, which is precisely the run you most want to inspect.

core/storage.py — persistence (165 lines)

One abstract interface, two implementations: InMemoryStorage for fast tests, SQLiteStorage for real runs. SQLite was chosen because it is in the Python standard library — a trace file is a single portable artifact with no server to run, no container to start, and no credentials to manage.

Business value: zero operational cost to adopt. A developer can attach a trace file to a ticket the way they would attach a log.

adapters/openai.py — the instrumentation (301 lines)

The largest and most commercially significant module. It is a transparent proxy: instrumented methods are traced, everything else is delegated via __getattr__. Three properties distinguish it:

  • The openai package is never imported. Response fields are read structurally through a helper that handles both dictionaries and attribute objects. The consequence is Section 6.3.
  • Tool calls are validated before execution — the security boundary described in Section 6.4.
  • Model mistakes and code failures are separated. A hallucinated tool becomes a recorded error fed back to the model so the loop recovers; a tool that genuinely raises propagates unmodified so real bugs are not buried in a trace entry.

viewer/tree.py — the shared tree builder (52 lines)

Turns the flat, parent_id-linked event list back into a nested structure. Day 3 deliberately kept this logic out of the CLI on the grounds that a real UI would need it; Day 5's API server became that second consumer, so it moved here and both callers share one implementation rather than two that drift. It also re-attaches orphaned events as roots — an event pointing at a parent not present in the trace was previously dropped from the timeline entirely.

viewer/render.py — the terminal timeline (66 lines)

Presentation only, since the tree logic moved out. Returns a string rather than printing, so the CLI prints it, the API serves it, and tests assert on it directly.

viewer/server.py — the JSON API (103 lines)

A read-only FastAPI application over one trace database, exposing four endpoints: trace summaries, one trace flat as stored, one trace nested as a timeline draws it, and one trace as rendered text. Three decisions carry weight:

  • A connection per request. FastAPI runs synchronous endpoints in a worker threadpool, and SQLite rejects a connection used off its creating thread — so the tracer's single-connection design would fail intermittently here. This was verified as a real failure before being designed around, and the test issues repeated requests because a single one passes either way.
  • Read-only and loopback-bound by default. There are no write endpoints, and the server binds to 127.0.0.1 rather than all interfaces, because traces hold prompts and tool arguments verbatim. Exposing them on a network is an explicit operator decision.
  • A missing database is an error, not an empty result. Since connecting creates a schema, a typo'd path would otherwise serve a brand-new empty database and report zero traces — making a wrong path indistinguishable from an unused one.

cli.py — the interface (81 lines)

Two subcommands, list and view. view defaults to the most recently started trace, because the common need immediately after a run is "show me what just happened," not "let me look up an ID first." User errors (bad ID, empty database) exit with status 1 and a message on stderr rather than a traceback — they are expected conditions, not bugs.

examples/openai_agent.py — the proof (205 lines)

A complete tool-calling agent that runs offline against a scripted client with no API key, and switches to any of four real providers with a flag. The scripted conversation deliberately includes a hallucinated book_flight call, so the error path is visible in the very first trace a new user generates.

Business value: evaluation friction is near zero. A prospective user sees a real trace within a minute of cloning, without a credit card.

tests/ — the safety net (39 tests, 630 lines)

Tests outnumber the modules they cover, with the adapter carrying the most (14) — appropriate, since it holds the security-sensitive logic. They run against hand-written fakes with no SDK installed, which is both faster and more honest: mocking the vendor SDK's types would only assert against our own assumptions about them.

6Key Engineering Decisions

Five decisions shaped the product. Each is presented as the trade-off it actually was.

6.1 A flat event schema, not a class hierarchy

Decision: one Event type with a free-form payload dict, rather than LLMCallEvent, ToolCallEvent, and so on.

Why: the schema must survive two pressures a class hierarchy fights — round-trip serialization to SQLite and JSON, and adapters that describe the same concept differently (LangChain's callback payloads look nothing like raw OpenAI function-calling payloads). A flat schema means one table and one serializer.

Cost, stated plainly: weaker type safety. An LLM event and a tool event are both just "an Event with a dict." Constructor functions recover some of that at the call site without subclasses.

6.2 A proxy, not a monkeypatch

Decision: trace_openai(client, tracer) returns a wrapper, rather than globally patching the SDK's method.

Trade-off: a proxy cannot trace a client constructed somewhere you cannot reach. In exchange it is explicit about what is traced, nests correctly when two tracers are alive simultaneously, and does not break when the vendor reorganizes SDK internals — a real maintenance liability for any monkeypatching library.

6.3 Target the wire format, not the vendor

Decision: never import openai; read response fields structurally.

Initially a testing convenience, this turned out to be the project's most valuable decision. When cost pressure prompted a move off OpenAI, the migration required zero adapter changes — xAI, Groq, and Ollama all expose OpenAI-compatible endpoints, so swapping a base_url was the entire effort. Tracing, token capture, tool dispatch, and schema validation were verified unmodified against all three.

Provider portability converts a vendor lock-in risk into a procurement lever. The same agent code, and the same trace format, run against a frontier model in production and a free local model in development — and the traces are directly comparable, which makes "is the cheaper model good enough?" an empirical question rather than an argument.

6.4 The tool schema is a security boundary

This decision came out of a pre-publication review and closed a genuine vulnerability class. Dispatching a tool means calling a Python function with arguments the model chose. Two problems followed from splatting that JSON straight into a callable:

The vulnerability

A Python function may have parameters its published schema never offered the model:

def read_file(path, allow_absolute=False):  # schema declares only `path`

Without a check, a prompt-injected tool call setting allow_absolute=True reaches that parameter and defeats a control the developer believed was private. The schema shown to the model is the contract; anything outside it must be unreachable.

The fix: arguments are restricted to the properties the schema declares, and then checked against the function's actual signature. Both come free — the schemas were already passed to create(tools=…), so the proxy records them and applies them automatically.

A deliberate non-behaviour: an unknown schema means skip the check, not deny everything. Denying would break every caller who never passed one. And the check bounds argument names, not values — validating that a path stays inside an allowed directory remains the tool's own job. Overstating this control would be worse than not having it.

6.5 Model errors versus code errors

A consistent rule, applied at two levels:

SituationHandlingReasoning
Model names a tool that does not existErrored span, fed back to the modelThis is agent behaviour NeuroTrace exists to reveal. It belongs in the trace, not in a traceback — and the loop can recover from it.
Model passes an argument the schema never offeredErrored span, fed back
Model misspells or omits a required argumentErrored span, fed back
Tool raises while executingPropagates unmodifiedThis is ordinary application code failing. Swallowing it would hide a real bug behind a trace entry.
Tool arguments are malformed JSONKept verbatim under _rawUnparseable arguments are the symptom; a decode error would destroy the evidence.

The split is now cleanly "wrong call" (recorded, recoverable) versus "broken tool" (raised). Before the Day 4a review, that line was drawn accidentally at whichever statement happened to throw first.

8Competitive Advantages

8.1 Near-zero adoption and exit cost

One construction line changes. The agent loop is untouched, which means the decision to adopt is cheap and — equally important to a cautious buyer — the decision to remove it is cheap too. There is no framework to migrate onto and no proprietary format to escape.

8.2 Data never leaves the machine

Traces are a local SQLite file. For teams handling regulated, confidential, or personal data, this is not a feature preference — it is the difference between usable and prohibited. Hosted competitors require shipping prompts and tool payloads to a third party.

8.3 Vendor-independent by construction

Because the adapter targets the wire format rather than a vendor SDK, provider migration costs a base_url. Traces from different providers share one format and are directly comparable — which turns model selection into a measurement rather than a debate.

8.4 Security posture as a differentiator

Most agent frameworks call fn(**model_chosen_args) without validation. NeuroTrace treats the published tool schema as an enforced boundary, and it does so by default because the schemas were already in the request. For a security reviewer, the trace also functions as an audit log: every tool invocation with every argument, including the ones that were rejected.

8.5 Extensibility already paid for

The flat schema, the adapter interface, and the read-time tree construction were designed for a second adapter and a second consumer before either existed. Adding Anthropic support is a sibling module, not a rewrite; adding a graphical viewer reuses the existing tree-builder.

8.6 A visible standard of engineering judgement

docs/architecture.md records not only what was decided but what was considered and rejected, and why. Two examples: a run-level error event was rejected because it double-counts what the span already recorded; renaming the module to openai_compatible.py was rejected because the accuracy gain does not justify a breaking import change. For a project intended to be public, this reasoning is itself a credential.

9Limitations & Risks

Read this section first if you are evaluating adoption

NeuroTrace is a pre-release library at Day 4 of a 7-day plan. The capture path is solid; several properties expected of production observability infrastructure are absent. They are listed here in descending order of consequence, not descending order of comfort.

9.1 Streaming responses record wrong data — silently

Calling create(stream=True) returns an iterator rather than a completed response. The adapter records an empty response body and a duration that measures only time-to-first-chunk. It does not error; it produces a trace that looks valid and is not.

Severity: high, because silent wrongness is worse than failure, and streaming is the default in most user-facing agents. Fix: wrap the iterator to accumulate deltas and close the span at the end — scheduled, and scoped as its own day of work.

9.2 Traces are unencrypted plaintext of everything the agent said

Prompts, responses, tool arguments, and tool results are written to SQLite verbatim. That is precisely what makes a trace useful — and it means a trace of a run whose system prompt embeds an API key, or whose tool arguments carry personal data, is now a plaintext copy of that secret on disk.

There is no redaction hook. Until there is, the operational controls are: keep .db files out of version control (already in .gitignore); treat a trace file as being as sensitive as the conversation it recorded, and read it before attaching it to a bug report; and place trace files where file permissions are appropriate when tracing production runs.

Severity: high for any regulated deployment. This is the single largest blocker between the current state and production use.

9.3 Persistence happens once, at the end of the run

Events accumulate in memory and are written when the Tracer block exits. Two consequences: a process killed hard (SIGKILL, OOM, power loss) loses the entire trace, including the crash you wanted to inspect; and a long-running agent holds every prompt and response in memory for its full duration. A run of thousands of steps with large payloads has an unbounded memory footprint.

Severity: medium. Incremental flushing is a natural extension — the storage layer already upserts, so re-saving an in-progress trace updates rather than duplicates.

9.4 Synchronous and single-threaded only

Three related gaps, one of which was closed on the server side in Day 5:

  • No async. The proxy wraps a synchronous create(). An AsyncOpenAI client returns a coroutine the proxy neither awaits nor recognises — much production agent code is async.
  • Capture is not thread-safe. The parent-tracking stack is an ordinary list with no thread-local isolation. Two agent threads sharing a tracer will interleave their stacks and produce a corrupted parent tree. Unchanged.
  • Connection not shareable — addressed for reads The SQLite connection is created once in the constructor and Python's driver rejects cross-thread use. This was confirmed as a live failure and designed around in the API server, which opens a connection per request. The underlying SQLiteStorage object is still single-thread, so the constraint remains for any other concurrent consumer.

Severity: medium-high — capture still bounds the addressable use cases to single-threaded synchronous agents, which is a real fraction of the market but not the majority of production deployments.

9.5 The flagship viewer does not exist

The product is described, in its own README, as "Chrome DevTools for agent runs." The terminal renderer delivers the structure but not the interaction — no expanding a step to read the full prompt, no filtering, no comparison. Section 3.3's tree is currently the entire visual product.

Day 5 removed the dependency rather than the gap: the JSON API the UI will consume now exists and serves a pre-nested tree, so Day 6 is a rendering exercise rather than a rendering and data-plumbing exercise.

Severity: medium for utility, high for adoption and demo value. This remains the largest gap between the stated positioning and the shipped artefact.

9.6 Prompt capture is lossy by design

The messages list is flattened into a single string for the event payload. This is adequate for reading a timeline and insufficient for replaying a request — the structure needed to reconstruct and re-run the exact call is discarded. If replay or regression-testing from traces becomes a goal, the payload format must change.

9.7 Coverage gaps in what is considered an "event"

  • The decision event type is declared in the schema but never emitted anywhere. The reasoning-step capture the project's own framing promises is not implemented.
  • retry events must be recorded manually. Retries performed inside the vendor SDK are invisible, so a trace can understate both latency and token spend.
  • Errors raised between spans — in the agent's own orchestration code — are not recorded as events. The trace still saves, but with an unexplained gap.

9.8 Scale and performance are uncharacterised

  • Improved Summary listing no longer loads every event of every trace. list_trace_summaries() is a single aggregate query with counts, used by both the CLI and the API's list endpoint. list_traces() retains the full-load behaviour, and the CLI's default "most recent trace" path still uses it to select a single row.
  • Tracing overhead has never been measured. It is expected to be small relative to network latency, but "expected" is not "measured," and no benchmark exists.
  • There is no sampling, retention policy, or size cap. A trace database grows without bound.

9.9 Ecosystem and interoperability

  • One protocol supported. Only the OpenAI wire format. Anthropic's distinct token fields and content-block shape need a sibling adapter. LangChain, LlamaIndex, and CrewAI — named in the project's own framing — have no adapter at all, so the "framework-agnostic" claim is currently architectural rather than demonstrated.
  • No OpenTelemetry export. Traces cannot flow into existing observability infrastructure, which for many organisations is a hard requirement rather than a nice-to-have.
  • Single-user, local-only. No server, no authentication, no sharing model. A trace is shared by sending someone a file.
  • Tests use hand-written fakes. A defensible choice, and it does mean no test exercises a real vendor SDK object. A silent upstream response-shape change would not be caught by CI.

9.10 Strategic risk

The clearest commercial risk is not technical. Agent observability is an actively contested category with funded entrants (LangSmith, Langfuse, Braintrust, Arize) and vendor-native tooling improving steadily. NeuroTrace's defensible positions are local-first data custody and true framework independence — both real, and both requiring the Days 5–7 work to be visible to anyone evaluating it.

11Why We Are Doing This

11.1 The bet on where the difficulty moves

Building an agent is now easy; a working loop is an afternoon's work. Trusting an agent remains hard. As agents move from demos into paths where they spend money, touch customer data, and take actions with consequences, the binding constraint shifts from capability to verifiability. Tooling that answers "what did it actually do, and why?" becomes infrastructure rather than convenience. NeuroTrace is a position on that shift.

11.2 Why local-first, specifically

The hosted platforms are capable, and for many teams they are the right answer. But a meaningful segment cannot use them at any price, because their prompts and tool payloads cannot leave their infrastructure. That segment is currently served by print(). Serving it with a zero-dependency local library is a narrow, defensible position that does not require out-competing a funded platform on features.

11.3 Why framework-agnostic, specifically

Every agent framework ships its own tracing, and every one of them ties observability to a framework choice that teams are still actively revisiting. A trace format that survives switching frameworks — or writing the loop by hand against a raw SDK — is worth more over time than one that does not, because it is the artefact that accumulates history.

11.4 Why build it in this order

Core, then adapters, then viewer. A viewer with no real data to display is a mockup; an adapter with nowhere to send data is a stub. Building the recording equipment first meant every subsequent layer was validated against real captured traces rather than imagined ones. The visible cost is that the most demo-able component arrives last — a real trade, made deliberately.

11.5 What success looks like

  • Near term: a developer diagnoses an agent failure from a trace instead of re-running with print statements, and it takes minutes rather than hours.
  • Medium term: a team compares four providers on latency, token spend, and tool-calling reliability using identical code and directly comparable traces.
  • Long term: the trace becomes the unit of record for agent behaviour — the thing attached to a bug report, reviewed in a security audit, and diffed between releases.
The honest summary

NeuroTrace is a well-architected pre-release library that has solved the hard part — capturing causally-linked agent execution with near-zero adoption cost and genuine provider independence — and has not yet built the part that makes it obviously valuable to someone who has not read the code. The engineering judgement on display is above the norm for a project this young. The gap between its positioning and its shipped surface is real, documented in Section 9, and closable in the three days already planned.

12.3 Glossary

TermMeaning
AgentA program that uses a language model in a loop to accomplish a goal, calling tools along the way.
LLM callOne round trip to the model. Costs tokens and takes time — the dominant contributor to both cost and latency.
Tool callThe agent invoking an external capability — a function, an API, a database query — rather than the model itself.
TraceThe complete recorded history of one agent run, start to finish.
EventOne recorded occurrence within a run. A trace is an ordered list of events.
SpanAn event with a duration — it opens, work happens, it closes. LLM calls and tool calls are spans.
PayloadThe type-specific content an event carries; for an LLM call, the prompt, response, and token counts.
AdapterCode translating a specific framework's or vendor's conventions into NeuroTrace's event format.
TracerThe component that wraps the agent loop and does the recording — a black-box flight recorder.
Wire formatThe shape of the request and response on the network, independent of any particular vendor's SDK. What the adapter actually targets.
ProxyA wrapper object standing in for another, intercepting some calls and passing the rest through untouched.
Tool schemaThe JSON description of a tool shown to the model — its name, purpose, and permitted parameters. In NeuroTrace, an enforced boundary.
Hallucinated callThe model requesting a tool that does not exist, or arguments that were never offered.
UpsertInsert-or-update. Used so re-saving an in-progress trace refreshes rows rather than duplicating them.
